核心挑战：为什么这不仅仅是 Prompt Engineering？
普通的 Prompt 经常会遇到以下痛点，而你的项目就是要通过微调（Fine-tuning）解决它们：

格式崩溃： 模型偶尔输出错误的 JSON 语法（少个逗号、引号未闭合）。

幻觉填充： 文本里没提到的字段，模型会瞎编一个。

模式偏离（Schema Drift）： 无法严格遵守复杂的嵌套模式。

长文本截断： 8B 模型在处理超长文档时容易丢失中间信息。

输入来源

5）推理层
上线架构建议做成两段式：

阶段 A：候选段落召回
先用规则从长财报里筛选相关段落：

“归属于上市公司股东的净利润”
“经营活动产生的现金流量净额”
“风险提示”
阶段 B：结构化抽取
把召回段落拼成紧凑上下文，再交给微调模型输出 JSON。

为什么这样做
因为财报很长，直接整篇抽取：

成本高
容易漏中间信息
容易引入无关噪声
这一步用轻规则反而是加分项，不是“土办法”。
